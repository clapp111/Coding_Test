### 기록

1. **슬라이딩 윈도우 + 해시맵 카운팅**
    
    이 문제는 배열의 연속성을 탐색하는 특징을 지니고 있다. 따라서 슬라이딩 윈도우 기법을 사용하는 것이 최적화하는데 유용하다. 
    
    슬라이딩 윈도우는 오른쪽 포인터를 확장하면서 종류를 채우고, 다 채워졌으면 왼쪽 포인터를 줄여서 길이를 최소화하는 방식으로 진행된다. 범위가 미끄러지듯 변하기 때문에 슬라이딩 윈도우라고 부르는 것 같다.
    
    원소의 포함여부는 단순히 bool 타입으로 하기보단 int 타입으로 카운팅해야 한다. 가능한 구간 내에 원소가 여러개 있을 수 있기 때문이다.
    
2. **고쳐야 할 점**
    
    처음에는 다음과 같은 방식의 풀이법을 구상했다.
    
    1. 가능한 구간까지 확장한다(start와 end 포인터 생성)
    2. start 포인터를 불가능한 구간 직전까지 땡긴다.
    3. 이렇게 만들어진 길이를 이전 최소 길이와 비교하여 갱신한다.
    4. 다음 시작 구간은 start + 1에서 시작한다.
    5. 1~5를 반복한다.
    
    이 방식의 단점은 가능한 구간까지 확장할 때, 즉 매 시작점마다 카운트 맵을 새로 만들고 끝까지 다시 스캔하는 방식이라, 효율성 테스트에서 계속 실패했다. → 윈도우를 재사용하지 못함
    
    또한 모든 종류를 포함했는지 확인하려고 맵 전체를 매번 검사했는데, 이런 로직은 불필요한 오버헤드를 유발했다.
    
    도저히 안되겠어서 다른 사람의 tistory에서 슬라이딩 윈도우가 정석이라는 걸 확인했다.
    

### 제출 코드

```cpp
#include <string>
#include <vector>
#include <set>
#include <unordered_map>

using namespace std;

vector<int> solution(vector<string> gems) {
    set<string> s(gems.begin(), gems.end());
    unordered_map<string, int> um;
    int total = s.size();
    int n = gems.size();

    int best_left = 0;
    int best_right = n - 1;
    int min_length = n;
    int left = 0;
    for(int right = 0; right < n; right++) {
        um[gems[right]]++;

        while(um.size() == total) {
            if(right - left < min_length) {
                min_length = right - left;
                best_left = left;
                best_right = right;
            }
            um[gems[left]]--;
            if(um[gems[left]] == 0) {
                um.erase(gems[left]);
            }
            left++;
        }
    }
    return {best_left + 1, best_right + 1};
}
```
